# PerfMon Architecture Documentation

PerfMon is built as a full-stack web application, following a modular and layered architecture to ensure scalability, maintainability, and clear separation of concerns.

## High-Level Overview

The system consists of three main components:

1.  **Client-Side Monitored Application**: The target web application where the PerfMon JavaScript snippet is embedded. It collects performance data and sends it to the PerfMon Backend.
2.  **PerfMon Frontend**: A React-based Single-Page Application (SPA) that provides the user interface for managing applications, pages, and visualizing performance reports.
3.  **PerfMon Backend**: A Node.js/Express TypeScript API that handles user authentication, CRUD operations for applications/pages, ingests performance data, processes it, and serves aggregated reports.
4.  **PostgreSQL Database**: Stores all application data, user information, and raw/aggregated performance metrics.

## Backend Architecture (Node.js/Express/TypeScript)

The backend follows a layered architecture with distinct responsibilities:

*   **`server.ts`**: The entry point, initializes the Express app and starts the server.
*   **`app.ts`**: Configures the Express application, middleware (CORS, Helmet, Body Parser, Custom Logging, Rate Limiting, Error Handling), and registers all API routes. Also handles database initialization.
*   **`config/`**:
    *   `env.ts`: Loads environment variables, providing a centralized configuration for the application.
    *   `winston.ts`: Configures Winston for structured logging across different log levels and transports (console, file).
    *   `data-source.ts`: TypeORM data source configuration for connecting to PostgreSQL.
*   **`entities/`**: TypeORM entity definitions (`User`, `Application`, `Page`, `PerformanceMetric`). These define the database schema and relationships.
*   **`database/`**:
    *   `migrations/`: Contains database migration scripts (generated by TypeORM CLI) for evolving the schema.
    *   `seeds/`: Scripts to populate the database with initial data (e.g., admin users).
*   **`middleware/`**:
    *   `auth.ts`: Implements JWT-based authentication for user-facing APIs and API Key authentication for performance data ingestion. Also includes authorization checks (e.g., `authorizeApplicationOwner`).
    *   `error.ts`: Centralized error handling middleware to catch errors, log them, and send consistent error responses. Includes a 404 handler.
    *   `logger.ts`: Custom middleware for logging HTTP requests and specific events (like performance data ingestion).
    *   `rateLimit.ts`: Configures `express-rate-limit` for different API endpoints to prevent abuse and ensure service stability.
    *   `cache.ts`: Implements an in-memory caching mechanism (`node-cache`) for read-heavy report endpoints.
*   **`routes/`**: Defines API endpoints for different resource domains (`auth`, `users`, `applications`, `pages`, `performance`, `reports`). Each route points to specific controller functions.
*   **`controllers/`**: Handle incoming HTTP requests. They parse request data, call appropriate service methods, and format responses. They should be thin and focus on HTTP concerns.
*   **`services/`**: Contain the core business logic. They interact with the TypeORM repositories to perform database operations, apply business rules, and orchestrate complex tasks. They are decoupled from HTTP specifics, making them easily testable.
*   **`utils/`**: Helper functions for common tasks like JWT generation/verification, password hashing, and input validation.
*   **`types/`**: Custom TypeScript type definitions.

### Key Backend Decisions:

*   **TypeORM**: Chosen for strong typing benefits and robust migration system with PostgreSQL.
*   **JWT Authentication**: Standard for secure, stateless user authentication.
*   **API Key Authentication**: For high-volume, potentially anonymous client-side data submission, providing a simple and efficient way to identify the source application.
*   **Structured Logging (Winston)**: Essential for monitoring, debugging, and auditing in a production environment.
*   **Centralized Error Handling**: Improves consistency and reduces boilerplate.
*   **Rate Limiting**: Protects against denial-of-service attacks and ensures fair usage.
*   **Caching**: Improves performance of reporting endpoints by reducing database load for frequently requested data.
*   **Dynamic Page Creation**: The system can automatically create new `Page` entries if `pageName` is provided in performance metric ingestion and no matching page exists. This simplifies setup for new pages.

## Frontend Architecture (React/TypeScript)

The frontend is a React Single-Page Application designed for a responsive and intuitive user experience.

*   **`index.tsx`**: Entry point for the React application, sets up the `AuthProvider` and `App` component.
*   **`App.tsx`**: Defines the main routing structure using `react-router-dom`, including public and private routes.
*   **`context/AuthContext.tsx`**: Uses React Context API for global state management related to user authentication (login status, user data, JWT token).
*   **`pages/`**: Top-level components representing different views/pages of the application (e.g., `Login`, `Dashboard`, `ApplicationDetails`, `PageDetails`, `NotFound`).
*   **`components/`**: Reusable UI components:
    *   `layout/`: Components for overall page structure (e.g., `Navbar`).
    *   `ui/`: Generic UI elements (e.g., `Button`, `Input`, `Modal`).
    *   `charts/`: Specialized components for data visualization using `Recharts` (`LineChart`, `PieChart`).
*   **`api/`**: Contains Axios configurations and functions for interacting with the backend API.
    *   `axiosConfig.ts`: Sets up the Axios instance, including base URL, default headers, and an interceptor for attaching JWT tokens to requests and handling 401 errors.
*   **`hooks/`**: Custom React hooks for encapsulating reusable logic (e.g., `useAuth`).
*   **`utils/`**: Frontend-specific utility functions.
*   **`types/`**: Shared TypeScript interfaces for data structures used across the frontend and potentially with the backend.
*   **`styles/tailwind.css`**: The main CSS file, utilizing Tailwind CSS for utility-first styling.

### Key Frontend Decisions:

*   **React + TypeScript**: Provides a robust framework for building interactive UIs with strong type safety.
*   **React Router DOM**: For declarative routing within the SPA.
*   **Context API for Auth**: A lightweight and effective solution for managing global authentication state without over-engineering.
*   **Axios Interceptors**: Simplifies API calls by automatically attaching authentication headers and handling common error responses.
*   **Tailwind CSS**: Enables rapid UI development with a consistent design system.
*   **Recharts**: A powerful and flexible charting library for visualizing performance data.

## Database (PostgreSQL)

*   **Relational Model**: Data is structured into `Users`, `Applications`, `Pages`, and `PerformanceMetrics` tables with clear foreign key relationships.
*   **Indexes**: Crucial indexes are defined on `performance_metrics` (`application_id`, `metric_type`, `timestamp`) and (`page_id`, `metric_type`, `timestamp`) to optimize data retrieval and aggregation queries, which are the backbone of the reporting functionality.
*   **Data Types**: Appropriate data types are used (e.g., `uuid` for IDs, `double precision` for metric values, `timestamp` for dates).
*   **Scalability Considerations (Future)**: For extremely high data volumes, techniques like table partitioning by time, materialized views for common reports, or even data warehousing solutions could be considered.

## Client-Side Performance Snippet

The JavaScript snippet for client-side applications is minimal and leverages modern browser APIs:

*   **`PerformanceObserver`**: Used to efficiently collect Core Web Vitals (FCP, LCP, CLS) without polling.
*   **`navigator.sendBeacon`**: Preferred for sending performance data to the backend because it's non-blocking and continues to send data even after the page is unloaded, improving data reliability.
*   **User/Session Tracking**: Uses a `localStorage` item to generate a consistent `userSessionId` for tracking individual user journeys anonymously.
*   **Dynamic Page Name**: Sends `document.title` and `window.location.href` as `pageName` and `url` respectively, allowing dynamic creation of pages on the backend if not pre-defined.

This architecture aims for a balance between comprehensive functionality, performance, and ease of development, making it suitable for an enterprise-grade performance monitoring solution.
