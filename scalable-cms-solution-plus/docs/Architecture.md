# CMS System Architecture

This document outlines the architectural design of the Content Management System.

## 1. High-Level Overview

The CMS system follows a **microservices-like architecture** with a clear separation of concerns, even if deployed as a single C++ backend executable for simplicity in this project. It consists of a C++ backend serving a RESTful API, a PostgreSQL database for persistent storage, a Redis instance for caching and session management, and a React-based frontend application.

## 2. Component Breakdown

### 2.1. Frontend (React Application)

*   **Technology**: React.js, React Router, Axios.
*   **Purpose**: Provides the user interface for content consumers and content creators/administrators.
*   **Interaction**: Communicates with the C++ Backend exclusively via RESTful API calls. Manages client-side routing, UI state, and user experience.
*   **Deployment**: Served as static assets (HTML, CSS, JS) via Nginx in a Docker container.

### 2.2. Backend (C++ Drogon Application)

*   **Technology**: C++17/20, Drogon Web Framework, `spdlog`, `jwt-cpp`, `libbcrypt`, `hiredis`.
*   **Purpose**: The core business logic layer. Exposes a RESTful API for all CMS operations.
*   **Architecture**:
    *   **Controllers**: Handle incoming HTTP requests, parse inputs, and orchestrate responses. They delegate business logic to services.
    *   **Services**: Encapsulate business logic (e.g., `AuthService`, `TokenService`, `CacheService`). They interact with the database and other external services.
    *   **Models**: Represent database entities (User, Post, Category). Generated by Drogon's ORM based on database schema.
    *   **Middleware (Drogon Filters)**: Intercept requests for cross-cutting concerns like authentication (`AuthMiddleware`, `AdminMiddleware`), rate limiting, logging, and error handling.
    *   **Utilities**: Helper functions for tasks like password hashing (`PasswordHasher`), logging (`Logger`).
    *   **Configuration**: `AppConfig` manages application settings loaded from JSON files and environment variables.
*   **Key Features Implemented**:
    *   **API Endpoints**: Full CRUD for Users, Posts, Categories.
    *   **Authentication & Authorization**: JWT-based access control, user roles (Guest, User, Admin, Editor).
    *   **Logging**: `spdlog` for structured, multi-sink logging (console and file).
    *   **Error Handling**: Consistent JSON error responses for API failures.
    *   **Caching**: `CacheService` with `hiredis` for Redis interactions (e.g., JWT blacklisting, potential data caching).
    *   **CORS**: Configured to allow communication from the frontend.

### 2.3. Database (PostgreSQL)

*   **Technology**: PostgreSQL.
*   **Purpose**: Persistent storage for all application data (users, posts, categories, etc.).
*   **Schema**: Defined in SQL migration scripts (`.sql` files). Includes tables for users, categories, posts, with appropriate relationships and indexes.
*   **ORM**: Drogon's built-in ORM is used for database interactions from the C++ backend, abstracting raw SQL queries.
*   **Migrations**: Handled by SQL scripts, typically executed by Docker's `docker-entrypoint-initdb.d` or a manual process.

### 2.4. Cache (Redis)

*   **Technology**: Redis.
*   **Purpose**: In-memory data store used for:
    *   **JWT Blacklisting**: Storing invalidated JWTs (e.g., on logout) to prevent their reuse.
    *   **Rate Limiting**: (Planned/extensible feature) Storing request counts per IP/user.
    *   **Session Management**: (Not explicitly implemented as full sessions, but JWTs are stateless; can be used for persistent sessions if needed).
    *   **General Caching**: Potentially caching frequently accessed data (e.g., popular posts).

### 2.5. Build & Deployment (Docker, CMake)

*   **CMake**: Used to manage the C++ backend build process, including external dependencies (Drogon, spdlog, hiredis, jwt-cpp, gtest).
*   **Docker**: Each major component (backend, database, Redis, frontend) is containerized using Dockerfiles.
*   **Docker Compose**: Orchestrates the multi-container application, making local development and single-host deployment straightforward. It defines how services interact, volumes, ports, and environment variables.

## 3. Data Flow Example: User Login

1.  **Client (Frontend)**: User enters credentials and clicks login.
2.  **Frontend**: Sends a `POST /api/v1/auth/login` request with email and password to the backend.
3.  **Backend (AuthController)**: Receives the request.
4.  **AuthService**: Validates input, queries `PostgreSQL` via Drogon's ORM to find the user by email.
5.  **PasswordHasher**: Verifies the provided password against the stored hash from the database.
6.  **TokenService**: If credentials are valid, generates an `accessToken` and a `refreshToken` (JWTs) with specific expiry times.
7.  **Backend (AuthController)**: Returns `200 OK` with `userId`, `accessToken`, `refreshToken`, and `role` in the response body.
8.  **Frontend**: Stores tokens (e.g., in `localStorage`) for subsequent authenticated requests.

## 4. Security Considerations

*   **Authentication**: JWT-based, with refresh tokens to minimize access token exposure. Access tokens are short-lived.
*   **Authorization**: Role-based access control implemented via Drogon filters (middleware) and checks within controllers.
*   **Password Hashing**: `bcrypt` is used for strong, one-way password hashing with configurable salt rounds.
*   **SQL Injection**: Drogon's ORM and parameterized queries inherently protect against most SQL injection attacks.
*   **CORS**: Explicitly configured for trusted origins.
*   **Input Validation**: Performed in controllers and services.
*   **Sensitive Data**: JWT secret, DB credentials, etc., are managed through environment variables and configuration files, not hardcoded.
*   **Logging**: Tracks security-relevant events (e.g., failed login attempts).

## 5. Scalability and Reliability

*   **Stateless Backend**: The C++ backend is stateless (using JWTs), allowing horizontal scaling by adding more instances behind a load balancer.
*   **Database**: PostgreSQL is robust and supports replication and clustering for high availability and read scaling.
*   **Caching (Redis)**: Offloads database queries and enables faster token validation, also scalable.
*   **Asynchronous Operations**: Drogon is an asynchronous framework, capable of handling many concurrent connections efficiently.
*   **Containerization**: Simplifies scaling and deployment to cloud platforms.

## 6. Future Enhancements

*   **Rate Limiting**: Implement a robust rate-limiting middleware using Redis.
*   **Object Storage**: Integrate with S3-compatible storage for media assets.
*   **Content Versioning**: Extend Post model and logic for content revisions.
*   **Full-Text Search**: Integrate with Elasticsearch or similar.
*   **Websockets**: For real-time features (e.g., notifications).
*   **GraphQL API**: Offer an alternative to REST for more flexible data fetching.
*   **Advanced UI**: More comprehensive admin dashboard, rich text editor, media library.
*   **Telemetry**: Integrate with Prometheus/Grafana for detailed metrics and dashboards.
*   **Robust Migration Tool**: Use a dedicated C++ migration library instead of raw SQL scripts.
