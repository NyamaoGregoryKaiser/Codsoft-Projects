# Enterprise-Grade Project Management System (Full-Stack)

This repository hosts a full-stack, enterprise-grade project management system, designed for high performance, scalability, and maintainability. It includes a robust API backend built with FastAPI, a dynamic frontend using React, and a PostgreSQL database with Redis for caching and rate limiting. The system emphasizes clean architecture, comprehensive testing, and modern DevOps practices.

## 1. Features

*   **User Management**: Register, Login, View Profile, Update Profile.
*   **Authentication & Authorization**: JWT-based authentication, role-based access control (Admin/Regular User).
*   **Project Management**: Create, Read, Update, Delete projects.
*   **Task Management**: Create, Read, Update, Delete tasks within projects.
*   **Comment System**: Add comments to tasks.
*   **Data Validation**: Robust input validation using Pydantic.
*   **Caching**: Redis-backed caching for frequently accessed data and rate limiting.
*   **Rate Limiting**: Protect API endpoints from abuse.
*   **Comprehensive Error Handling**: Global exception handling.
*   **Structured Logging**: Detailed logs for monitoring and debugging.
*   **Database Migrations**: Alembic for schema evolution.
*   **Containerization**: Docker for isolated and reproducible environments.
*   **CI/CD**: GitHub Actions for automated testing and deployment.
*   **Interactive API Docs**: Swagger UI (OpenAPI) and ReDoc automatically generated by FastAPI.

## 2. Architecture Overview

### Backend (FastAPI)
The backend is built with FastAPI, a modern, fast (high-performance), web framework for building APIs with Python 3.7+ based on standard Python type hints.

*   **Framework**: FastAPI
*   **ORM**: SQLAlchemy with AsyncIO support
*   **Database Driver**: `asyncpg`
*   **Authentication**: PyJWT
*   **Password Hashing**: Passlib
*   **Caching/Rate Limiting**: Redis via `aioredis`
*   **Request Validation**: Pydantic
*   **Testing**: Pytest, httpx
*   **Database Migrations**: Alembic

**Key Components:**
*   **`main.py`**: Entry point of the FastAPI application, registers routers, middleware, and exception handlers.
*   **`app/core/`**: Core utilities like configuration, database session management, security (JWT, password hashing), caching, and custom exceptions.
*   **`app/schemas/`**: Pydantic models for request body validation, response serialization, and data definitions.
*   **`app/models/`**: SQLAlchemy ORM models defining the database schema.
*   **`app/crud/`**: Functions encapsulating database operations (Create, Read, Update, Delete) for each model, abstracting away raw SQLAlchemy queries.
*   **`app/api/deps.py`**: Dependency injection functions for database sessions, current user authentication, and rate limiting.
*   **`app/api/v1/`**: API routers, organized by resource (auth, users, projects, tasks, comments), handling HTTP requests and responses.
*   **`app/middleware/`**: Custom FastAPI middleware for logging, error handling, etc.

### Frontend (React)
The frontend is a Single Page Application (SPA) built with React, providing a dynamic and responsive user interface.

*   **Framework**: React
*   **Styling**: Tailwind CSS
*   **State Management**: React Context API
*   **HTTP Client**: Axios
*   **Routing**: React Router DOM

**Key Components:**
*   **`src/index.js`**: Application entry point.
*   **`src/App.js`**: Main component, sets up routing.
*   **`src/api/api.js`**: Axios instance for API communication, handles JWT token injection.
*   **`src/context/AuthContext.js`**: Provides authentication state and functions (login, logout) to the entire application.
*   **`src/pages/`**: Components representing full-page views (Login, Dashboard, Project Details, etc.).
*   **`src/components/`**: Reusable UI components (buttons, forms, cards, navigation).
*   **`src/styles/index.css`**: Tailwind CSS imports and custom styles.

### Database (PostgreSQL)
A robust and highly-performant relational database used for persistent storage of all application data.

### Caching & Messaging (Redis)
Redis is used as an in-memory data store for:
*   **Caching**: Storing frequently accessed data to reduce database load and improve response times.
*   **Rate Limiting**: Tracking API call counts per user/IP to prevent abuse.

## 3. Getting Started

### Prerequisites

*   **Docker & Docker Compose**: Essential for containerized development and deployment.
    *   [Install Docker](https://docs.docker.com/get-docker/)
*   **Python 3.9+** (for running scripts outside Docker, e.g., Alembic migrations)
*   **Node.js & npm** (for frontend development outside Docker)

### Local Development Setup

1.  **Clone the repository:**
    ```bash
    git clone https://github.com/your-username/project-management-system.git
    cd project-management-system
    ```

2.  **Create `.env` file:**
    Copy `.env.example` to `.env` in the root directory and fill in your desired values.
    ```bash
    cp .env.example .env
    # Open .env and customize, e.g., DATABASE_URL, SECRET_KEY, REDIS_URL
    ```
    Ensure `DATABASE_URL` matches the service name in `docker-compose.yml` if running locally:
    `postgresql+asyncpg://user:password@db:5432/app_db`

3.  **Build and run Docker containers:**
    ```bash
    docker-compose up --build -d
    ```
    This will start:
    *   `db` (PostgreSQL)
    *   `redis` (Redis)
    *   `backend` (FastAPI app, accessible at `http://localhost:8000`)
    *   `frontend` (React app, accessible at `http://localhost:3000`)

4.  **Access the applications:**
    *   **Frontend**: `http://localhost:3000`
    *   **Backend API**: `http://localhost:8000`
    *   **Backend API Docs (Swagger UI)**: `http://localhost:8000/docs`
    *   **Backend API Docs (ReDoc)**: `http://localhost:8000/redoc`

### Database Migrations

**IMPORTANT**: Ensure the `db` container is running (`docker-compose up -d db`).

1.  **Connect to the backend container for Alembic:**
    ```bash
    docker-compose exec backend bash
    ```

2.  **Run migrations:**
    ```bash
    alembic upgrade head
    exit # to exit the container
    ```

    *To generate a new migration (after changing `app/models/`):*
    ```bash
    docker-compose exec backend bash
    alembic revision --autogenerate -m "Add new feature table"
    # Review the generated migration script in alembic/versions/
    alembic upgrade head
    exit
    ```

### Seeding Data

After migrations, you might want to populate the database with some initial data.

1.  **Connect to the backend container:**
    ```bash
    docker-compose exec backend bash
    ```

2.  **Run the seed script:**
    ```bash
    python seed_db.py
    exit # to exit the container
    ```
    This will create an admin user (email: `admin@example.com`, password: `adminpassword`) and some sample projects and tasks.

## 7. Deployment Guide

The `docker-compose.yml` file is suitable for local development. For production deployment, you would typically use:

*   **Kubernetes**: For container orchestration and scalability.
*   **Docker Swarm**: A simpler alternative to Kubernetes.
*   **Cloud Providers (AWS ECS/EKS, Azure AKS, Google Cloud GKE/Cloud Run)**: Managed container services.

**General Production Deployment Steps:**

1.  **Build Docker images**: Ensure your `Dockerfile`s are optimized for production (multi-stage builds, smaller images).
2.  **Environment Variables**: Securely manage environment variables (e.g., AWS Secrets Manager, Kubernetes Secrets). Do NOT hardcode secrets.
3.  **Database Management**: Use a managed PostgreSQL service (e.g., AWS RDS, Azure Database for PostgreSQL).
4.  **Redis**: Use a managed Redis service (e.g., AWS ElastiCache, Azure Cache for Redis).
5.  **Reverse Proxy**: Deploy a reverse proxy (Nginx, Caddy) in front of your FastAPI and React applications for SSL termination, load balancing, and static file serving.
6.  **Monitoring & Logging**: Integrate with a robust monitoring system (Prometheus, Grafana, Datadog) and a centralized logging solution (ELK stack, Splunk, CloudWatch).
7.  **CI/CD Integration**: Automate deployment via your CI/CD pipeline.

## 8. CI/CD Pipeline

A basic CI/CD pipeline is configured using GitHub Actions (`.github/workflows/main.yml`). This pipeline performs:

*   **Backend**:
    *   Lints Python code (Flake8)
    *   Runs unit and integration tests (Pytest)
    *   Builds the Docker image (Optional, for push to registry)
*   **Frontend**:
    *   Installs dependencies
    *   Lints JavaScript code (ESLint)
    *   Builds the React application (Optional, for push to static hosting)

**To extend for full deployment:**
*   Add steps to push Docker images to a container registry (e.g., Docker Hub, AWS ECR).
*   Add steps to deploy to a cloud provider (e.g., `aws cli` commands, `kubectl` commands).

## 9. Configuration

All sensitive and environment-specific configurations are managed via environment variables. The `app/core/config.py` module loads these variables, providing a centralized configuration object.

**Key environment variables:**
*   `DATABASE_URL`: PostgreSQL connection string.
*   `SECRET_KEY`: Used for JWT signing. **MUST be a strong, random string in production.**
*   `ALGORITHM`: JWT algorithm (e.g., HS256).
*   `ACCESS_TOKEN_EXPIRE_MINUTES`: JWT token expiry time.
*   `REDIS_URL`: Redis connection string.
*   `FASTAPI_ENV`: `development` or `production`. Influences logging level.

## 10. Logging & Monitoring

The backend implements structured logging using Python's standard `logging` module.
*   Logs are outputted to `stdout`, making them easily collectable by container orchestration platforms (Docker, Kubernetes) and centralized logging solutions.
*   A custom middleware (`app/middleware/log_request.py`) logs incoming requests and outgoing responses, including execution time.
*   Error handling includes logging stack traces for internal server errors.

For production, integrate with tools like ELK stack (Elasticsearch, Logstash, Kibana), Datadog, or Grafana Loki for centralized log aggregation and monitoring.

## 11. Error Handling

*   **Custom Exceptions**: `app/core/exceptions.py` defines custom exceptions for specific business logic errors (e.g., `CredentialException`, `HTTPException`).
*   **Global Exception Handlers**: `main.py` registers handlers for `HTTPException` and `RequestValidationError` to return consistent JSON error responses.
*   **Middleware**: An error handling middleware ensures that even unhandled exceptions are caught, logged, and returned with a generic 500 error to clients (to avoid leaking sensitive internal details).

## 12. Caching

*   **Redis Backend**: `app/core/cache.py` provides an `aioredis` client for interacting with Redis.
*   **Decorator for Caching**: A simple caching decorator (`@cached`) is demonstrated in `app/api/v1/users.py` (e.g., for getting user by ID). This can be extended for more complex caching strategies (e.g., cache invalidation).

## 13. Rate Limiting

*   **Redis Backend**: Rate limiting data (request counts, timestamps) is stored in Redis.
*   **FastAPI Dependency**: `app/api/deps.py` includes a `rate_limiter` dependency that can be applied to any API endpoint. It uses a sliding window algorithm to limit requests per client.
*   **Configuration**: The rate limit is configurable (e.g., 5 requests per 10 seconds).
